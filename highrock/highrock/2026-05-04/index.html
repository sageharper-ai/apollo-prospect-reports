Executive Summary

HighRock's on-page foundation is in better shape than typical baseline audits suggest — but with a consistent set of gaps that compound into AI Overview invisibility and inconsistent SERP previews. The Drupal Metatag module is already installed and configured for Open Graph tags across every audited page. Image alt text is on every image (a rare positive — most audits find big gaps here). Canonicals are correct site-wide.

What's missing across the site: Twitter Card tags (configurable on the existing Metatag module), JSON-LD structured data (requires the Schema.org Metatag submodule install), and explicit robots meta. Per-page issues: homepage and /work both lack an H1 (one of the two most common cause-of-invisibility patterns), several titles are too short to carry value-prop weight (15–19 chars where 50–65 fits), and the /services meta description gets SERP-truncated at 343 chars. Most fixes are short — the heavy lift is the Schema.org Metatag module install + per-payload configuration.

The Drupal-config nuance — why this is a smaller dev lift than it looks

Existing Metatag module IS deployed and configured per-page. Apollo verified: every audited page emits og:site_name, og:type, og:url, og:title, og:description. That means the foundation work is done — Twitter Card sub-fields are configurable inside the same module, no new install needed.

One module install required: Schema.org Metatag (drupal/schema_metatag) — extends the existing Metatag module with structured-data field types. Composer install + enable + per-content-type config. Apollo can walk the dev lead through it in a 30-min sync if useful.

What this cycle is, and isn't

Is: Manually-prepared on-page recommendations for HighRock's homepage + /services + /work + /about. Each draft is written against current page state captured today + HighRock's brand voice from on-site copy ("Elevate your brand," "Strategy · Creativity · Innovation," team-and-mountain framing on /about). Isn't: Service-line sub-pages (8 confirmed at /services/strategic-marketing, /services/creative, etc. — recommended for cycle 2); individual case-study pages under /work (cycle 2 or 3); content production for new pages (separate scope); a full-site audit (cycle 1 covers 4 pages of ~60+).

Drupal deployment — Schema.org Metatag module path (recommended)

  1. Composer install + enable. composer require drupal/schema_metatag · drush en schema_metatag (or admin → Extend → enable). Module pairs with the existing Metatag module already deployed for OG tags.
  2. Configure the schema types Apollo's bundle uses. Admin → Configuration → Search and metadata → Metatag → Settings → enable: schema_organization, schema_professional_service, schema_collection_page, schema_about_page, schema_breadcrumb_list, schema_creative_work.
  3. Set defaults. Admin → Configuration → Metatag → Add default metatag → Global scope for Organization root payload (deploys on every page); per-content-type for ProfessionalService (homepage + /services), CollectionPage (work), AboutPage (about).
  4. Map Drupal field tokens to schema fields. Each schema field accepts Drupal tokens (e.g., [node:title], [node:field_service_description]) so payloads stay in sync with content edits.
  5. Twitter Card configuration (no new install — uses existing Metatag module). Same Metatag config UI: enable Twitter Card sub-fields, set defaults per content type.
  6. Validate. Live URLs → Google's Rich Results Test → confirm "Eligible for rich results" on each schema type. Test all 5 payload types before bulk-applying.
  7. Submit affected URLs to Google Search Console for re-indexing once schema is live. Re-index window typically 3–14 days for surface-level inclusion; AI Overview eligibility builds over weeks.

Drupal deployment — Twig template fallback (only if module doesn't fit)

If specific schema types aren't supported by the Schema.org Metatag module's field set, JSON-LD can drop directly into themes/custom/highrock/templates/html.html.twig:

Twig fallback is more code per template but gives finer control if Drupal-side field mapping isn't viable. Apollo's recommendation: use the module path. Twig is the escape hatch.

Recurring quarterly

Drupal sites without OTTO management run on a slower cadence than SA-managed sites — quarterly is realistic. Each cycle picks up a new content tier or refreshes outdated drafts. The Quarterly Impact Report follows: pages affected since last cycle, GSC impression/click delta, schema validation snapshot, content-gap recommendations queued for next cycle.

Out-of-scope for this cadence

  • Content production for new pages — separate engagement.
  • Authority work (link building, digital PR) — separate engagement.
  • Drupal performance optimization (Core Web Vitals) — overlaps with on-page but warrants its own scope; available on request.
  • Site migration (Drupal version upgrade, theme rebuild) — separate engagement.
